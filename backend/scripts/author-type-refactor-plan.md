# Author Type Refactor Plan

## Original prompt

> I want a 3rd category: AI generated. I changed my mind ‚Äî now I want to show to the general public that those 50 users generated via AI (we generated the pictures, names and poems with the help of AI) are valid users to show to people, and they will be able to interact with them in the future. On the other hand, I still want to fake some users until I do have real users.
>
> We have 2 fields in the DB (origin and fake) and I don't know if that's the best approach. What I want:
>
> - In the UI, in the filter/dropdown, people can select: "All" authors, "User users" and "AI users"
> - People visiting the site will have a clear picture of what kind of author wrote a poem (3 possibilities: a famous writer, a common user of the website, and an AI user generated by us)
>
> Internally I want to organise like this:
> - For **famous authors**: note the origin of where the poems were copied from (e.g. Poetry Foundation or other sites) ‚Äî internal info only
> - For **non-famous users**:
>   1. Real users that come and register on the website
>   2. Fake users created temporarily (no real users yet, want to give the impression of activity ‚Äî eventually deleted)
>   3. AI-generated users, proudly shown as AI, given their own personality
>
> Assign the **7 manually created users** (admin-impersonated) ‚Üí fake user category.
> Assign the **50 seed-script users** ‚Üí AI category.

---

## Current state

### Author model fields used for classification

| Field | Values in use | Purpose |
|---|---|---|
| `origin` | `'user'`, `'famous'`, `'Poetry Foundation'` | Mixed: public category + internal source |
| `fake` | `true` / `false` | Was used to mark all non-real users as fake |

### Problems with current structure

1. `origin` mixes **public concept** (famous vs user) with **internal source** (Poetry Foundation).
2. `fake: true` is used for both temporary fake users and AI-generated users ‚Äî no way to distinguish them.
3. There is no way to mark a user as "AI" in the DB.
4. The 50 AI users and the 7 manual fake users currently look identical in the DB (`origin: 'user'`, `fake: true`).

### How the 7 vs 50 users are identified

- **50 AI-generated users**: created by `backend/scripts/seed-fake-users.js`. All have `email` ending in `@fakemail.com`. These are the ones to mark as `type: 'ai'`.
- **7 manual fake users**: created by the admin via the impersonation UI before the seed script ran. They also have `fake: true` but do NOT have `@fakemail.com` emails. These are the ones to mark as `type: 'user'` + `fake: true`.

## Proposed schema

### Author model ‚Äî new classification fields

| Field | Type | Values | Purpose |
|---|---|---|---|
| `type` | String | `'famous'` \| `'user'` \| `'ai'` | **Public-facing category** ‚Äî used in UI filter |
| `source` | String | `'poetry-foundation'` \| `'manual'` \| `null` | **Internal** ‚Äî tracks where famous poems were sourced from |
| `fake` | Boolean | `true` / `false` | **Internal** ‚Äî marks users presented as user but not real registrations (may be AI-assisted). To be deleted once real users arrive. |

Drop: the old `origin` field on Author (replaced by `type` + `source`).

### Public filter logic

| UI label | DB filter |
|---|---|
| All | _(no filter)_ |
| User users | `type: 'user'` |
| AI users | `type: 'ai'` |
| Famous writers | `type: 'famous'` |

### Internal-only combinations

| Situation | `type` | `source` | `fake` |
|---|---|---|---|
| Famous poet (all currently from Poetry Foundation) | `'famous'` | `'poetry-foundation'` | `false` |
| Real registered user | `'user'` | `null` | `false` |
| Temporary fake user (manually or AI-assisted, presented as user) | `'user'` | `null` | `true` |
| Openly declared AI user | `'ai'` | `null` | `false` |

### Poem model

Poems have their own `origin` field (currently `'user'`, `'famous'`, `'Poetry Foundation'`). This field is used in poem-level filtering (`/api/poems?origin=famous`).

Proposed values after migration:

| Poem origin | Meaning |
|---|---|
| `'famous'` | Written by a famous poet |
| `'user'` | Written by a user user (real or fake ‚Äî internal distinction lives on Author) |
| `'ai'` | Written by an AI-generated author |

> Note: currently both `'famous'` and `'Poetry Foundation'` exist on poems. Both collapse to `'famous'` during migration. The source detail moves to `Author.source`.

## Migration plan

### Step 1 ‚Äî Update Author model (`src/models/Author.js`)

- Add `type: { type: String, enum: ['famous', 'user', 'ai'] }`
- Add `source: String` (nullable)
- Keep `fake: Boolean`
- Keep `origin` temporarily (for the migration script to read from)

### Step 2 ‚Äî Write migration script (`scripts/migrate-author-types.js`)

```
For each Author:
  if origin === 'Poetry Foundation'  ‚Üí type = 'famous', source = 'poetry-foundation'
  if origin === 'famous'             ‚Üí type = 'famous', source = 'manual'
  if origin === 'user' AND email ends with '@fakemail.com'  ‚Üí type = 'ai',    fake = false
  if origin === 'user' AND fake === true (non-fakemail)     ‚Üí type = 'user',  fake = true
  if origin === 'user' AND fake === false                   ‚Üí type = 'user',  fake = false
```

### Step 3 ‚Äî Update Poem `origin` values

```
For each Poem:
  look up its Author (via authorId)
  if author.type === 'famous'  ‚Üí poem.origin = 'famous'
  if author.type === 'user'   ‚Üí poem.origin = 'user'
  if author.type === 'ai'      ‚Üí poem.origin = 'ai'
```

### Step 4 ‚Äî Update `register.js`

New users registered on the site get: `type: 'user'`, `fake: false`, `source: null`

### Step 5 ‚Äî Update controllers

- `authors.js` `buildFilter()`: replace `origin` filter with `type` filter
- `poems.js`: update `origin === 'famous'` logic to use `{ $in: ['famous'] }` (simplified ‚Äî no more two values)
- Update `POST /api/poems` to set `poem.origin` from author's type (not hardcoded `'user'`)

### Step 6 ‚Äî Update frontend

- `AdminFields.tsx`: the "Origin" dropdown options change to `famous` / `user` / `ai`
- Dashboard author filter dropdown: add "AI users" option
- Poem cards / author pages: show a badge or label when `type === 'ai'`

### Step 7 ‚Äî Drop old `origin` field from Author

After migration is verified in staging, remove `origin` from Author documents and from the schema.

---

## What does NOT change

- The 7 manual fake users remain in the DB ‚Äî they just get `type: 'user'`, `fake: true` instead of `origin: 'user'`, `fake: true`
- The admin impersonation UI (`AdminFields.tsx`) still works ‚Äî it fetches `type=user&fake=true` authors
- The 50 AI users get `type: 'ai'` and their `fake` flag is cleared to `false`
- Famous poets are unaffected beyond the field rename

---

## Decisions (open questions resolved)

1. **AI users always separate** ‚Äî the "User users" filter never includes AI authors.
2. **Filterable only for now** ‚Äî no AI badge on poem cards or list items yet.
3. **Profile top bar** ‚Äî AI author pages show `"AI generated"` in parentheses next to the username. No other UI difference for now.
4. **Single famous source** ‚Äî all famous authors in the DB come from the Poetry Foundation Kaggle dataset. No other source exists at this point. `source: 'poetry-foundation'` is the only value needed. `source: 'manual'` is dropped from scope.
